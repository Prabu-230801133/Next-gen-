Creating a File: You can create any type of file (e.g., Example.md) and add content to it.

Staging a File: Use the command git add <file_name> to stage a file, indicating that you want to include it in the next commit and "git add ." for staging all changes in the current directory.

Checking Status: The git status command helps you see the current state of your repository, including which files are staged and ready to be committed.

Committing a File: Use the command git commit -m "commit message" to commit changes from the staging area to the local repository. This creates a snapshot of the file state.

Commit Message: It's important to include a message that explains why you are committing the file, e.g., "add example.md to demonstrate the git process".

Checking Status: After committing, use git status to verify that everything is committed. This will show that your local repository is now ahead of the remote repository.

Push Changes: Use the command git push to push changes from your local repository to the remote repository.

Verify Changes: After pushing, you can verify that the changes were successful by checking the remote repository's web interface.

Commit Message: The commit message set earlier will be visible in the remote repository, confirming the changes.

Pulling Changes: Use the command git pull to update your local repository with changes from the remote repository.

Remote Ahead of Local: The remote repository may have newer files or changes that need to be synchronized with your local repository.

Commit Changes: Ensure any changes made in the remote repository are committed and then pulled to your local environment.

Initialize a Repository: Use git init to create an empty Git repository in your local folder.

Add and Commit Files: Stage files with git add . and commit them using git commit -m "initial commit".

Link to Remote Repository: Connect your local repository to a remote one using git remote add origin <URL>.

Push Changes: Use git push to push your local commits to the remote repository, and follow any error messages to set up branches if necessary.

Tracking Changes: Git can track changes within files, not just add new ones. You can commit changes to a file using the same process as committing a new file.

Using git diff: The git diff command allows you to see what has changed in your files before committing. It provides a clear overview of modifications, highlighting new additions in green.

Reviewing Staged Changes: If you have already added files to the staging area but still want to review the changes, you can use git diff --cached to see the staged changes before committing.

Staging Area Rollback: Use git restore --staged <file> to remove a file from the staging area and move it back to the working directory.

Complete Rollback: Use git restore <file> to completely undo changes in the working directory, reverting the file to its previous state.

All Files Rollback: Use git restore --staged . to remove all staged files at once and git restore . to undo all changes in the working directory.

Viewing Commit History: Use git log to see the commit history, including the commit ID, author's name, date, and commit message.

Filtering and Exploring: Commands like git show, git log -p, git log --oneline, and git log --grep='phrase' help filter and explore commit details.

Graphical Representation: Use git log --graph to visualize branch history.

Remote Repository: You can also view commit history through the remote repository's web interface for a visual breakdown of changes.

Listing Old Commits: Use git log --oneLine to list old commits with shortened commit IDs.

Checking Out Older Versions: Use git checkout <commit ID> to switch to an older version of the project.

Detached Head State: Be aware that checking out an older commit puts you in a detached head state, which is not ideal for making changes.

Returning to Current State: Use git checkout <branch name> (often main) to return to the current state of your project.

Reverting Changes: Use git revert <commit ID> to undo changes from a specific commit without rewriting history.

Commit Messages: After reverting, Git opens an editor to modify the commit message, which documents the revert action.

Maintaining History: Reverting creates a new commit that undoes the previous changes, ensuring the history remains intact and clear.
